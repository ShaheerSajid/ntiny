async-trap epc when ID stage is empty

← bug catalog · Bug E · fixed · commit a9b6e72

Symptom
Linux 7.0 with Sstc enabled crashed in terminate_walk with ra = 0xfffff000 — the high half of PAGE_MASK, not a real return address.
Files touched
design/core/interrupt/src/interrupt_ctrl.sv

What the kernel saw

At a function epilogue, the kernel did lw ra, 28(sp) to restore the saved return address. The value loaded was 0xfffff000 — the kernel PAGE_MASK constant for RV32 — which is not a valid code address. The next instruction ret jumped there, instantly faulted, and the kernel oopsed.

Probing the prologue side of the same function showed the sw ra, 28(sp) store had successfully committed a real return address. Something between prologue and epilogue zapped the saved ra in the stack frame.

What actually happened

The bug fires in a narrow window: an async timer interrupt arrives at the same cycle a function-prologue instruction reaches IE, immediately after a context switch where the ID stage is empty. In ntiny the ID stage shows up as pc_id = 0 when no instruction is buffered.

The trap controller's pc_for_async chooser (= what to put into sepc) was: 

pc_for_async = async_use_branch ? brcvr
             : async_use_ie     ? pc_ie
             : pc_id != 0       ? pc_id     // skip past
             : pc_ie             // fallback when ID empty
             : pc_out;

When an async-trap fires with ID empty, the chooser hit the pc_id == 0 condition and fell through to pc_ie — which is the IE-stage PC, the instruction we're currently executing. The kernel then resumed at that PC after sret and re-executed it. The IE op had already retired; re-running doubled its effect. For a function prologue that was addi sp, sp, -16, the second execution decremented sp a second time, putting sp + 28 off the actual saved ra slot and into adjacent memory that happened to hold 0xfffff000.

The plain-language version

Imagine the kernel parks its bicycle in slot 4 of a rack. The fire alarm rings. When it comes back, the kernel parks the bike in slot 4 a second time — but slot 4 is occupied, so it ends up in slot 0. Later when the kernel reaches for the bike at slot 4, it grabs whatever someone else left there — in this case a road sign that reads 0xfffff000. It tries to ride the road sign and crashes.

The fix

When the ID stage is empty, the IE op has already retired (its writeback is in flight to IMEM/IWB). The right resume PC is "the instruction after the IE op" — i.e. pc_ie + insn_len. Insn-length-aware via c_valid_ie_i so RVC and RV32 both work: 

// interrupt_ctrl.sv
wire [31:0] pc_ie_next   = pc_ie_i + (c_valid_ie_i ? 32'd2 : 32'd4);
wire [31:0] pc_for_async = async_use_branch ? brcvr
                         : async_use_ie     ? pc_ie
                         : (pc_id_i != 0)   ? pc_id
                         : (pc_ie_i != 0)   ? pc_ie_next  // skip Z
                         :                    pc_out;

Now the chooser correctly skips past the just-retired IE op when the ID stage is freshly empty, instead of accidentally landing on the IE op's PC and re-executing it.

Verification

Linux 6.6 boot with Sstc enabled now reaches the buildroot login prompt cleanly. v7.0 with Sstc reaches /init after the related fixes (A, B) land.
