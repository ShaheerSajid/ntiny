async-trap collides with taken branch at IE

← bug catalog · Bug B (xas_load "Layer 3") · fixed · commit 9766f90

Symptom
Linux 7.0 boot reaches /init (with bug A also fixed), then crashes with Oops - store (or AMO) address misaligned inside next_uptodate_folio on the first start-stop-daemon page-fault.
Found at
sim cycle ~193.1M on Linux 7.0; FST window [193.08M, 193.12M]
Files touched
design/core/interrupt/src/interrupt_ctrl.sv, design/core/core_top/src/core_top.sv

What the kernel saw

An lr.w (load-reserved) inside next_uptodate_folio faulted with badaddr=c200e4de — an address whose low two bits are 10, i.e. word-misaligned. lr.w requires 4-byte alignment, so the trap is correct in spirit. The interesting question is how a misaligned address ended up there.

The faulting code looks something like this: 

c00c00d8: addi s8, a0, 28        # s8 = folio + 28 (= refcount addr)
c00c00dc: lr.w s9, (s8)          # <<< trap fires here

At the trap, a0 = 0xc200e4c2, which decodes as a sibling-encoded XArray entry (a special pointer with low bits set to indicate the slot is a redirect, not a real folio). XArray library code earlier in the call chain (xas_load) is supposed to canonicalise sibling pointers — turn them into real folio pointers — before they leak out. So a0 arriving as a sibling pointer means xas_load didn't run its canonicalisation step.

What actually happened

Inside xas_load, the canonicalisation runs only if a certain branch is taken: 

c0282e62: andi x15, x10, 3       # low 2 bits of a0
c0282e66: addi x15, x15, -2
c0282e6a: bnez x15, c0282e70      # if low2bits != 2 → fall through
c0282e6c: bltu x14, x10, c0282e7a # <<< the branch in question
c0282e70: ... fall-through path ...
c0282e7a: ... canonicalisation path ...

At the cycle of the trap-fire (FST sim_time 386 199 156), the bltu at c0282e6c was at IE, with the timer interrupt firing in the same cycle. The pipeline state: 

opA_forwarded_data  = 0x00001000     (= a4, from preceding lui correct)
opB_forwarded_data  = 0xc200e4c2     (= a0, the slot value, correct)
branch_taken        = 1              (a4 < a0, so bltu takes)
branch_taken_valid  = 0              (BPU correctly predicted T)
branch_target_addr  = 0xc0282e7a     (correct taken target)

interrupt_valid     = 1              (timer fires this cycle)
ecause              = 0x80000005     (S-mode timer)
epc captured        = 0xc0282e70     (!!!)

The branch was taken, the recovery target was correct, the BPU was on the same page — everything in the pipeline did the right thing. But the captured sepc was the fall-through PC of a branch that took. After sret, the kernel resumed at c0282e70, executed the "non-canonical" fall-through path, and returned the raw sibling pointer to xas_load's caller. That pointer flowed all the way to the lr.w — the sibling's low bits made the address misaligned and the alignment check faulted.

why the "use the recovery target" path missed

interrupt_ctrl had: 

async_use_branch = branch_taken_i;   // = bpu_mispredict
pc_for_async     = async_use_branch ? brcvr
                 : ...

branch_taken_i in interrupt_ctrl was wired to bpu_mispredict. When the BPU predicts correctly, mispredict = 0, so the chooser skipped brcvr and fell through. Mispredicted branches always ended up using brcvr, but correctly-predicted-TAKEN branches didn't — so the bug only fires when the BPU happens to be right.

The plain-language version

Picture a road with a fork. The pipeline reaches the fork (bltu), the branch evaluator says "take the right exit", the BPU agrees, and traffic redirects right — exactly as expected. Then a fire alarm rings (timer interrupt). The kernel writes down the address to come back to (sepc). By mistake it writes down the address "just past the fork sign on the straight road" (the fall-through PC) instead of "the start of the right-fork road" (the recovery target). When the alarm clears and the kernel returns, traffic resumes on the wrong road.

In all the other branch outcomes the address book had a special "branch detour" entry pointing at the right place. But for the "everyone agreed and it just took normally" case, no one had thought to fill that entry in.

The fix

Pass the architectural direction (output of branch_comp) into interrupt_ctrl as a separate signal, and OR it into the gate: 

// core_top.sv
.branch_taken_h_i (branch_taken == TRUE),

// interrupt_ctrl.sv
wire async_use_branch = branch_taken_i      // mispredict
                      | branch_taken_h_i;   // architecturally taken

Now brcvr is the sepc source for any IE-stage branch that's committing — whether mispredicted or not. brcvr already collapses all four predict×direction combinations into the architecturally-correct next-PC, so a single OR is sufficient.

Why this fix had to come bundled with bug A

When this fix was applied alone, the kernel got past the xas_load trap but trapped a few million cycles later inside event_create_dir with a refcount=0 WARN. Bug A turned out to be the same async-trap-vs-IE-stage race, just for AMOs instead of branches: the OLD bug here was masking it by truncating loop iterations on taken backward branches. With this fix in place, the loop runs to completion and reaches the buggy AMO. Both fixes ship together in commit 9766f90.
