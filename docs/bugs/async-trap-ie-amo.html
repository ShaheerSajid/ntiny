async-trap collides with AMO at IE

← bug catalog · Bug A · fixed · commit 9766f90

Symptom
Linux 7.0 boot stops with refcount_t: addition on 0; use-after-free WARN inside event_create_dir+0x232 while event_trace_init runs in a workqueue. The kworker dies and /init never gets scheduled.
Found at
sim cycle ~179.69M (sim_time 359 376 576) on Linux 7.0 boot
Files touched
design/core/amo_unit/src/amo_unit.sv, design/core/interrupt/src/interrupt_ctrl.sv, design/core/core_top/src/core_top.sv

What the kernel saw

Linux puts the refcount_t family of counters under a hard guard: incrementing a counter that is already zero is a use-after-free, period. If anyone tries it, the kernel fires a BUG_ON-style breakpoint (ebreak, cause=3) and dumps a stack trace.

The trace pointed cleanly at event_create_dir, which registers tracefs entries for every kernel trace event. The instruction is a single atomic increment on the event's refcount field: 

c00ae664: addi a0, s4, 40         # a0 = &event_call->refcount
c00ae668: li   a1, 1
c00ae66a: amoadd.w a5, a1, (a0)   # a5 = OLD value, *a0 += 1
c00ae66e: beqz a5, c00ae7dc       # if OLD was 0, jump to WARN call
c00ae7dc: li   a1, 2              # WARN_SATURATE_INC
c00ae7e2: jalr ra, refcount_warn_saturate

So a5 = 0 after the AMO. Either the in-memory refcount really was zero, or the AMO returned a wrong value for some other reason. The smoking gun was buried in the pipeline.

What actually happened

The cycle the amoadd.w reached the IE (execute) stage, the S-mode timer interrupt fired in the same cycle. We captured the pipeline state in an FST waveform and walked it cycle-by-cycle: 

sim_time   pc_ie        interrupt_valid   ie_flush   amo_active   amo_dbus_*
358999156  c00ae66a     1                 1          0            (no transition)
358999158  squashed     0                 0          0            (no transition)
358999160  c00ae66e     0                 0          0            (no transition)
                        ↑ AMO retire writeback fires: rd=a5, data=0

Three things matter:

  1. The AMO never accessed memory. amo_dbus_addr, amo_dbus_read, amo_dbus_write — all stayed flat. The flush_i input to amo_unit (driven by interrupt_valid) blocked its IDLE → AMO_READ transition.
  2. amo_active stayed 0 the entire time. amo_unit exposes active_o = (state == AMO_READ || state == AMO_WRITE), so a squashed-before-start AMO looks the same as no AMO at all.
  3. The IWB stage still retired the AMO instruction one cycle later, writing amo_unit.result_q (= 0, never updated by a memory read) into a5.

why the "skip" path picked the wrong PC

interrupt_ctrl picks sepc using: 

async_use_ie = amo_active | mmu_d_stall;
pc_for_async = async_use_branch ? brcvr
             : async_use_ie     ? pc_ie       // re-execute
             : pc_id != 0       ? pc_id        // skip past
             : ...

With amo_active = 0 and mmu_d_stall = 0, async_use_ie = 0, so the chooser fell through to pc_id (= the instruction right after the AMO). The kernel's sret resumed at c00ae66e with a5 = 0 — the spurious value the AMO had already written to the regfile. The next instruction is beqz a5, …, so the BUG fired.

The plain-language version

Imagine a clerk (the AMO unit) about to start an atomic read-modify-write on a counter. As they raise their pen to begin, the fire alarm rings (timer interrupt). They drop the work. But the desk's outbox still has a paper labelled "I did the work, here is what I read" — and that paper has zero on it because no work actually happened. The supervisor (the IWB stage) takes the paper at face value, files it (writes a5 = 0), and the kernel's next step (beqz a5) interprets that as "object was freed" and panics.

The correct response in this situation is "hold on, the AMO didn't actually run; resume back at the AMO after the alarm so the clerk can try again." That's what the fix does.

The fix

amo_unit exposes a new output that captures "an AMO is at IE waiting to start, but hasn't activated yet": 

// amo_unit.sv
output logic pending_o;
assign pending_o = (amo_op_i != NO_AMO_OP) && (state == IDLE);

This is high before the AMO transitions out of IDLE, and goes back low in the DONE state (memory write already committed). interrupt_ctrl ORs it into the "re-execute on async trap" gate: 

// interrupt_ctrl.sv
wire async_use_ie = amo_active_i | mmu_d_stall_i | amo_pending_i;

Now when the timer fires the same cycle as a fresh AMO at IE, async_use_ie = 1, sepc = pc_ie, and sret resumes at the AMO — the kernel re-runs the increment cleanly, with the correct OLD value, and beqz doesn't trigger.

The DONE-state exclusion is essential: an AMO that already wrote to memory must not re-execute, or the increment doubles. An earlier attempt at this fix gated on ie_amo_op != NO_AMO_OP and double-decremented an rwsem counter in chr_dev_init. Restricting pending_o to the IDLE state is what makes this safe.

Verification

After the fix, Linux 7.0 boot reaches Run /init as init process + busybox crond. Linux 6.6 reaches the buildroot login: prompt with the boot self-test reporting 19 PASS / 0 FAIL. The fix is not kernel-version-specific.
