Sstc IE→IMEM pipeline race

← bug catalog · Bug D · fixed · commit 2e803a1

Symptom
With Sstc (S-mode timer extension) enabled, an S-mode timer interrupt fires while an IE-stage instruction is committing to IMEM. The instruction commits AND its PC ends up in sepc. After sret the kernel re-runs the same instruction, doubling its effects.
Files touched
design/core/core_top/src/core_top.sv

What the kernel saw

Without Sstc, S-mode timer interrupts go through an SBI ecall to M-mode, which adds enough latency that a colliding interrupt basically can't happen. With Sstc the timer fires directly in S-mode — cycle-accurate — and starts hitting the pipeline in places SBI never could.

Concrete observed effect: the "just-committed" instruction's side effect (a register update or a store) happened twice. For example a sw ra, 28(sp) in a function prologue committed once normally, then again after sret — the second time corrupted because ra in the meantime had been overwritten by the trap-handler entry sequence.

What actually happened

In ntiny's pipeline, the IE-stage register-wall captures the IE-stage instruction's exec result on the cycle the instruction is allowed to retire. Async-trap entry has a window where:

  • The IE-stage instruction's writeback path has already latched its result — it WILL retire through IMEM and IWB.
  • Simultaneously, the trap controller decides sepc = pc_ie (the very instruction whose result is already in the writeback pipeline).

After sret, the kernel resumes at sepc — the instruction we already committed — and runs it again. The first commit's effect is permanent; the second re-execution adds another effect on top.

The plain-language version

Imagine the kernel writes a check to pay a bill. They sign and seal it. Just as the courier picks it up, the fire alarm rings. The kernel notes "I was about to write that check" in their to-do list, then runs to the alarm. When they come back, they read their to-do, see "write check", and write a second copy. The bill gets paid twice.

The fix is to recognise: the courier already left with the first check — so the to-do entry should say "done, move on", not "to-do".

The fix

Inject an IMEM bubble (a NOP that doesn't retire) on the cycle of trap entry, gated by interrupt_valid && ecause_csr[31] — the high bit of cause indicates an asynchronous interrupt. This stops the IE-stage instruction's writeback from propagating into IMEM that cycle, so the trap fires cleanly with the IE op uncommitted — sepc = pc_ie is then the right answer, and re-execution is safe.

Aftermath

Sstc is now safe to leave on in the device tree. The fix was bundled with the "narrow async-trap re-execute gate to AMO mid-FSM" commit (2e803a1), which together pushed Linux 6.6 RV32 SMP-N=1 boot through to userspace.

See also: bug A (AMO collision), bug B (branch collision), bug E (post-context-switch empty-ID epc). All four are different shades of the same problem: getting the right architectural sepc when asynchronous interrupts collide with in-flight pipeline state.
