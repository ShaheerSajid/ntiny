BPU vs IE-flush race

← bug catalog · Bug C (pty_init kernfs corruption) · fixed · commit 54cd2f7

Symptom
Linux 7.0 panics in pty_init with kernfs node corruption: a delete_node call dereferences a NULL child pointer.
Files touched
design/core/core_top/src/core_top.sv (BPU consumer / IE-wall arbitration)

What the kernel saw

pty_init populates kernfs entries for the pseudo-tty subsystem. One of the loops walks an in-progress kernfs tree and expected each parent to have a populated child list before it ran delete_node. The kernel saw a node where the child list pointer had been overwritten with garbage at some earlier cycle — and so when it dereferenced the "child" it hit a NULL or wild address and panicked.

What actually happened

The Branch Prediction Unit (BPU) speculates which way a branch will go, redirecting fetch ahead of the actual branch resolution. Sometimes the BPU and the IE-stage flush logic try to redirect fetch in the same cycle. ntiny has an arbiter that chooses between them.

In the failing run, a self-induced ie_flush killed an IE-stage bltu. The BPU's redirect (which had fired one cycle earlier on the predicted-T target) was sitting in the arbiter, waiting to be consumed. The arbiter granted the BPU. But the BPU's consumer (the "can I take this redirect?" signal) was reading a stale, registered version of mmu_i_stall, while the IF/ID stall signal was using the live (unregistered) version. They disagreed for one cycle. Result: the BPU thought it had been consumed, dropped the redirect, but the predicted target's first instruction had already been fetched and pushed into IMEM — and committed an unguarded store.

the leaked store

The speculative store wrote to a kernfs internal pointer field. No-one updated that field again before pty_init walked the tree, so it stayed corrupted until the loop tried to follow it.

The plain-language version

Two air-traffic controllers (the BPU's redirect and the branch-flush logic) both want to clear a runway in the same second. They disagree by one tick of the clock about whether weather is OK to land. One controller releases a plane onto the runway thinking it's clear. The other thinks the plane bounced off and went back to the holding pattern. The plane lands and unloads its cargo (a memory write), but the manifest says the plane never landed. Hours later the warehouse is filled with cargo no-one expected and the inventory manager (delete_node) walks into a wrecked stack.

The fix

When the IE-wall arbiter resolves a redirect-kind to RDR_BPU (the BPU got the win), capture the branch at the IE wall in the same cycle. The override clause in core_top.sv ensures the IE-stage bltu is still resolved into a taken-target redirect, even when the BPU and IE flush both agree the branch should be killed. That removes the one-cycle skew between "I'm consumed" and "I'm dropped".

Verification

After the fix, Linux 7.0 boots through pty_init cleanly and reaches userspace + busybox. The same fix unblocked the rest of v7.0 boot (it had been the cliff that earlier work could not get past).
